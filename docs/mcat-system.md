# MCAT System

Full documentation for the student-side **`/mcat`** feature — an MCAT **Biology**-only practice platform modeled on the precalc `/learn` + `/demo-practice` flows, but with its own isolated `mcat_*` tables, on-the-fly generation, embedding-based keyword tagging, and an AAMC-content-outline-grounded generator.

**Scope:** Biology only. Chemistry / Physics / Psych-Soc are stubbed "coming soon" in the UI. Adding a section is more taxonomy rows + a tag-prefix mapping + a content-outline entry — **no new code paths**. Do **not** expand beyond Biology unless explicitly asked.

> Everything MCAT lives under the `mcat_` table prefix and `apps/student/{app/mcat,app/api/mcat,components/mcat,lib/mcat*}`. It never touches the precalc `learn_*` / `problems` / `rag_examples` tables, so the two systems can't leak into each other.

---

## Taxonomy

Three tiers, all in `mcat_keywords` (`tier` column):
- **category** (`mcat_categories`) — 10 Biology categories (e.g. `mcat_biology_amino_acids_and_proteins`).
- **umbrella** keyword — 106 broad topics (e.g. `protein_separation_and_analysis`), `parent_keyword_id = null`.
- **in_depth** keyword — ~740 narrow, single-concept skills (e.g. `sds_page_separation_by_molecular_weight`), `parent_keyword_id =` the umbrella id.

Seeded from `mcat-keywords.txt` (two concatenated top-level JSON objects — a category/umbrella catalog + the amino-acids in-depth set) via `npm run seed:mcat`. The other 9 categories' in-depth keywords are generated by `npm run mcat:expand` (5–8 narrow keywords per umbrella, each targeting one fact/mechanism/distinction so a wrong answer isolates exactly what's misunderstood).

**Keyword targeting uses in_depth preferred, umbrella fallback** — `loadTargetKeywords(supabase, categoryIds)` in `lib/mcatTagging.ts` returns the in_depth keywords for each category, falling back to umbrella keywords only if a category has no in_depth tier.

## Pages (`apps/student/app/mcat/`)

| Route | Purpose |
|---|---|
| `/mcat` | Landing: section tabs (Biology active), General Practice card, 10 category cards (mastery bar + Practice/Flashcards/Quiz + "Explore topics →" into the browse page). Login/logout in header (`components/mcat/AuthButtons.tsx`). |
| `/mcat/[categoryId]` | **Browse / drill-down.** Whole-category actions, then each umbrella (expandable) with Practice/Flashcards/Quiz/Lesson, expanding to its in-depth keywords each with the same four actions. |
| `/mcat/[categoryId]/practice` | **The Duolingo-style loop** (see below). Honors `?umbrella=<id>` / `?keyword=<id>` scope params. |
| `/mcat/[categoryId]/quiz` | 8-question quiz (no mid-quiz reveal; review list at end with per-question FeedbackWidget). Scope params resolve to `keyword_ids`. |
| `/mcat/[categoryId]/flashcards` | Flip cards (tap to flip **both ways**) → Got it / Missed it / I didn't know. Scope params. |
| `/mcat/lesson/[keywordId]` | Standalone lesson page (works for any umbrella **or** in_depth keyword id). `?label=` for the header. |
| `/mcat/practice` | **General Practice** — hierarchical topic picker (category → umbrella → in_depth tri-state checkboxes), then the practice loop across the chosen `keyword_ids` / `category_ids`. |
| `/mcat/progress` | Mastery dashboard: per-category umbrella rows (implied score = avg of attempted children), expandable to in_depth rows with score bars, low-sample hints, "didn't know ×N" badges, mastered checks, and a "Lesson" link per keyword. |

**Scope param scheme** (the three action pages): none = whole category; `?umbrella=<id>&label=` = that umbrella's children; `?keyword=<id>&label=` = a single keyword. `useSearchParams` is wrapped in `<Suspense>` (Next 15 requirement — mirror `/login`).

## API routes (`apps/student/app/api/mcat/`)

| Route | Method | Purpose |
|---|---|---|
| `taxonomy` | GET `?session_id=` | Categories → flat `keywords` (back-compat) **and** nested `umbrellas[].children[]` with session scores + `implied_score`. |
| `practice-queue` | GET `?session_id=&category_id=[&umbrella_id=][&keyword_id=]` | Weakest-first keyword queue (cap 40, excludes mastered) + `review_pool`. `keyword_id` → single-keyword queue; `umbrella_id` → that umbrella's children. |
| `next-question` | POST | Serve/generate one question. Scope via `keyword_id` (single), `keyword_ids` (set), or `category_id`/`category_ids`. Optional `difficulty: 'easy'|'medium'|'hard'`. |
| `attempt` | POST | Record answer (or `dont_know`), update `mcat_student_keyword_states` via EMA, return `keyword_states` with `needs_lesson`. |
| `similar` | POST `{session_id, question_id}` | Generate a same-difficulty variant testing the same concept; insert with `parent_question_id`. |
| `quiz` | POST `{session_id, category_id, count?, difficulty?, mixed?}` | Assemble N questions (stored + generated). `mixed: true` spreads easy→medium→hard. |
| `flashcards` | POST `{session_id, category_id, count?, keyword_id?, keyword_ids?}` | Stored + generated flashcards, weakness-ranked. |
| `flashcard-attempt` | POST | Record got_it/missed_it/dont_know, update keyword states. |
| `lesson/[keywordId]` | GET | Cache-or-generate a 3–4 step micro-lesson (stored in `mcat_lessons`). |
| `feedback` | POST `{session_id, content_type, content_id, rating?, flagged?, flag_reason?}` | Rate/flag a question/flashcard/lesson. ≥2 flags → question/flashcard `status='flagged'` (stops serving); lesson deleted (forces regen). |

All routes use the service-role Supabase client and the `{ error, detail }` + 502 contract for generation failures (same as `learnGenerator.ts`).

## Practice loop (`/mcat/[categoryId]/practice`)

**Lessons are the LAST resort, not the opener.** Questions/flashcards are the diagnostic; a lesson only appears on demonstrated struggle or an explicit click.

Per-keyword flow:
1. **New keyword** (`total_attempts === 0 && score === null`) → **flashcard warm-up**: up to 2 keyword-targeted cards ("Quick warm-up"), graded, then questions.
2. Otherwise → straight to **questions**.
3. **Mastery-gated progression** (replaces a fixed block size): keep serving questions on the topic — biasing toward `similar`-question variants when the last answer was correct (~50%) so variants of one concept batch together — until **mastered** (`state === 'mastered'` OR client `topicCorrectStreak >= 3`) or a hard cap of **8** questions (un-mastered → re-queued once at the end). A header "mastery meter" (●●○) shows streak toward 3.
4. **Spaced review between topics**: on advancing, ~35% chance (if `review_pool` non-empty) to insert one review question from a previously-learned topic (prefers `spaced_review_due_at <= now`, else lowest score), badged "Review", before the next topic starts.
5. **Lessons** appear only via the amber struggle banner (2 consecutive wrong / dont_know / server `needs_lesson`, once per keyword) or the always-visible "Learn this" header button. After a lesson, return to the topic's questions (streak reset to 0).

**Difficulty selector** in the header — `Adaptive · Easy · Medium · Hard`, persisted to `localStorage["mcat_practice_difficulty"]`. Non-adaptive modes send `difficulty: <tier>` on every `next-question`.

## Generation (`apps/student/lib/mcatGenerator.ts`)

Model **`gpt-5.4-mini`**, JSON mode, no streaming, typed `McatGenError` → 502 (mirrors `learnGenerator.ts`). Exports: `generateMcatQuestions`, `generateSimilarQuestion`, `generateMcatFlashcards`, `generateMcatLesson`.

**Two grounding inputs injected into every generation user prompt:**
1. **Template cards** — `fetchTemplateCards()` (`lib/mcatTemplateCards.ts`) pulls real facts from the MileDown anki deck in `anki_cards`, matched by the category's `CATEGORY_TO_TAG_PREFIXES` tag prefixes + term overlap.
2. **AAMC content outline** — `outlineContextForCategory(categoryId)` (`lib/mcatContentOutline.ts`) returns the official AAMC content-category (1A–3B) scope + canonical topic list, so generated content stays in the scope/depth/terminology the real MCAT tests. Injected above the template-card block.

**Choice shuffling** — every generated MCQ (questions, similar, lesson check questions) passes through `shuffleChoices()` (Fisher–Yates, recomputes `correct_index`, skips on duplicate/≠4 choices). This kills the index-0 bias the LLM otherwise produces. Existing stored rows were fixed once via `npm run mcat:rebalance` (see scripts).

## Difficulty model

`difficulty` on `mcat_questions` is a continuous **[0.2, 0.9]** value, **LLM-set within a requested band** (so "hard" is genuinely hard — see [difficulty-scales.md](difficulty-scales.md) for the band definitions and how this differs from precalc's 1–5 / `estimated_difficulty` scales).

- **Generation:** `generateMcatQuestions` takes `targetDifficulty?: number` and/or `difficultyTier?: 'easy'|'medium'|'hard'`; the prompt describes EASY (single-step recall), MEDIUM (one inference), HARD (multi-step reasoning / two concepts / all-plausible distractors). `generateSimilarQuestion` inherits the source question's difficulty.
- **Adaptive target** (`next-question`): `base = 0.2 + avgStrength*0.6`, plus an escalation bump `min(0.15, 0.05 * maxConsecutiveCorrect)` over the scoped keywords, clamped to [0.2, 0.9]; the tier is derived from the target and both are passed to generation. An explicit `difficulty` body param overrides the adaptive target and biases stored-candidate selection toward that band (2× in-band multiplier on the difficulty-fit gaussian).
- **Candidate scoring** (stored questions): `weakness × difficultyFit(gaussian σ=0.2) × ratingNudge(0.7 + 0.3*(avg_rating/5))`, weighted-random pick from top 5.

## Mastery state & EMA (`mcat_student_keyword_states`)

One row per `(session_id, keyword_id)`. **Kept normalized (not a JSONB blob)** because every read needs per-keyword querying: weakest-N selection, umbrella implied-score averaging, `spaced_review_due_at` filtering, and atomic per-keyword upserts under concurrent attempts. Same rationale and shape as the precalc `learn_student_keyword_states`.

Per attempt, for each keyword in the question's `keyword_weights`:
- EMA via `updateStrengths` (`lib/practiceAlgorithm.ts`, α=0.12, weight-scaled, first-touch target 0.75/0.25). `dont_know` counts as incorrect and increments `dont_know_count`.
- `state = 'mastered'` when `score >= 0.8 && consecutive_correct >= 4`; on mastery it schedules `spaced_review_due_at` (via `computeNextReviewDate`, REVIEW_INTERVALS `[1,3,7,14,30,60]`) and bumps `spaced_review_count`. `needs_lesson` when `score < 0.35 && total_attempts >= 3`, or immediately on a `dont_know`. Else `in_progress`.
- Keyword ids not present in `mcat_keywords` are skipped (FK safety).

## Embedding & keyword tagging

`text-embedding-3-small`, stored as JSONB arrays. `npm run mcat:embed` runs three phases: keywords (`label. description`), questions (`stem | correct_choice`), flashcards (`front | back`). Questions and flashcards are **retagged** after embedding: cosine similarity vs the in_depth keywords of their category → top 4 with sim > 0.25 (fallback top 2), normalized to sum 1. `cosineSimilarity` + `tagByEmbedding` live in `lib/mcatTagging.ts`. New content is embedded+retagged inline on generation (try/catch — embedding failure never fails generation; the row is left for the batch script).

`npm run mcat:clean-unembedded` deletes any question/flashcard still missing an embedding (deletes dependent attempts first for FK safety) — guarded behind `--apply` (dry-run by default).

## Scripts (`scripts/`, run with `tsx`)

| Command | Script | Purpose |
|---|---|---|
| `npm run seed:mcat` | `seed-mcat-taxonomy.ts` | Seed `mcat_categories` + `mcat_keywords` from `mcat-keywords.txt`. `--dry-run`. |
| `npm run mcat:expand` | `expand-mcat-keywords.ts` | Generate 5–8 in_depth keywords per childless umbrella. `--dry-run`, `--limit N`, `--category <id>`. Idempotent (skips umbrellas that already have children). |
| `npm run mcat:embed` | `embed-mcat.ts` | Embed keywords/questions/flashcards + retag. `--dry-run`. |
| `npm run mcat:rebalance` | `rebalance-mcat-correct-index.ts` | One-time: shuffle stored questions' choices to fix index-0 bias. `--dry-run`. |
| `npm run mcat:clean-unembedded` | `delete-unembedded-mcat.ts` | Delete unembedded content. Dry-run unless `--apply`. |

> **Env quirk:** root `.env.local` has an **invalid** `OPENAI_API_KEY`; the valid one is in `apps/student/.env.local`. Scripts that call OpenAI override `process.env.OPENAI_API_KEY` from the app env (run them as `DOTENV_CONFIG_PATH=apps/student/.env.local npx tsx -r dotenv/config <script>` if needed). Supabase keys are valid in the root `.env.local`.
