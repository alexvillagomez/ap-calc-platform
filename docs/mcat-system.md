# MCAT System

Full documentation for the student-side **`/mcat`** feature — an MCAT **Biology**-only practice platform modeled on the precalc `/learn` + `/demo-practice` flows, but with its own isolated `mcat_*` tables, on-the-fly generation, embedding-based keyword tagging, and an AAMC-content-outline-grounded generator.

**Scope:** **Biology** + **Psych/Soc** (Psychological, Social, and Biological Foundations of Behavior, FC 6–10 / content categories 6A–10A — added 2026-06-26, see [mcat-psychsoc-taxonomy-2026-06.md](mcat-psychsoc-taxonomy-2026-06.md)). Chemistry / Physics remain stubbed "coming soon" in the UI. The two live sections share **all** code paths; a section is just `mcat_categories.section` + a `section` URL param (**default `biology`**, so Biology flows are unchanged) + tag-prefix/content-outline entries. Adding Chem/Physics would follow the same recipe. Do **not** expand beyond the two live sections unless explicitly asked.

> Everything MCAT lives under the `mcat_` table prefix and `apps/student/{app/mcat,app/api/mcat,components/mcat,lib/mcat*}`. It never touches the precalc `learn_*` / `problems` / `rag_examples` tables, so the two systems can't leak into each other.

## Taxonomy

Three tiers, all in `mcat_keywords` (`tier` column):
- **category** (`mcat_categories`) — 10 Biology categories (e.g. `mcat_biology_amino_acids_and_proteins`).
- **umbrella** keyword — 109 broad topics (e.g. `protein_separation_and_analysis`), `parent_keyword_id = null`. (Was 106; +3 = Lipid Structure, Enzyme Classification, Sensory Systems — see the 2026-06-26 AAMC audit, [docs/mcat-taxonomy-audit-2026-06.md](mcat-taxonomy-audit-2026-06.md).)
- **in_depth** keyword — 934 narrow, single-concept skills (e.g. `sds_page_separation_by_molecular_weight`), `parent_keyword_id =` the umbrella id. **The whole taxonomy is now 1043 rows (>1000), so any query loading all keywords MUST paginate (`lib/mathPagedQuery.ts` `fetchAllPages`)** — the taxonomy route and `loadTargetKeywords` do; an un-paginated `select` silently truncates at 1000. Visible keywords are `status='approved'` (new rows must be inserted/updated to `approved`, not `active`).

Seeded from `mcat-keywords.txt` (two concatenated top-level JSON objects — a category/umbrella catalog + the amino-acids in-depth set) via `npm run seed:mcat`. The other 9 categories' in-depth keywords are generated by `npm run mcat:expand` (5–8 narrow keywords per umbrella, each targeting one fact/mechanism/distinction so a wrong answer isolates exactly what's misunderstood).

**Keyword targeting uses in_depth preferred, umbrella fallback** — `loadTargetKeywords(supabase, categoryIds, opts?)` in `lib/mcatTagging.ts` returns the in_depth keywords for each category, falling back to umbrella keywords only if a category has no in_depth tier. It selects `order_index` and accepts `{ excludeIntros?: boolean }` — question-selection callers (`next-question`, `quiz`) pass `excludeIntros: true` to drop framing-only intro keywords (`order_index === -1`) from the candidate pool (see [Intro keywords](#intro-keywords--framing-only-order_index---1)).

## Auth gate & onboarding

**MCAT requires login.** Every page funnels through `getOrCreateMcatSession()` (`lib/mcatSession.ts`), which now gates on auth: if there is no `ap_calc_account_id` in localStorage it redirects to `/login?next=<current path>` and returns a never-resolving promise (the page stays in its loading state until navigation). After login, the `?next=` param (honored by `/login`) returns the student to where they were. Logging out (`AuthButtons`) clears the keys and sends them back through the gate. Because all 8 pages call the helper on mount, the whole feature is gated with no per-page code.

**Quick onboarding** — `components/mcat/McatOnboarding.tsx` is a 4-step first-visit overlay rendered on `/mcat`. It self-gates on `localStorage["mcat_onboarding_seen"]` and only shows to logged-in students (so there's no flash before the login redirect). "Skip" or finishing sets the flag.

All pages are **login-gated** (see above). 

| Route | Purpose |

| `/mcat/[categoryId]/quiz` | 8-question quiz (no mid-quiz reveal; review list at end with per-question FeedbackWidget). Scope params resolve to `keyword_ids`. |
| `/mcat/[categoryId]/flashcards` | Flip cards (tap to flip **both ways**) → Got it / Missed it / I didn't know. Scope params. |
| `/mcat/lesson/[keywordId]` | Standalone lesson page (works for any umbrella **or** in_depth keyword id). `?label=` for the header. |

| `/mcat/progress` | Mastery dashboard: per-category umbrella rows (implied score = avg of attempted children), expandable to in_depth rows with score bars, low-sample hints, "didn't know ×N" badges, mastered checks, and a "Lesson" link per keyword. |

## API routes (`apps/student/app/api/mcat/`)

| Route | Method | Purpose |
|---|---|---|
| `taxonomy` | GET `?session_id=` | Categories → flat `keywords` (back-compat) **and** nested `umbrellas[].children[]` with session scores + `implied_score`. |
| `practice-queue` | GET `?session_id=&category_id=[&umbrella_id=][&keyword_id=]` | Weakest-first keyword queue (cap 40, excludes mastered) + `review_pool`. `keyword_id` → single-keyword queue; `umbrella_id` → that umbrella's children. |
| `next-question` | POST | Serve/generate one question. Scope via `keyword_id` (single), `keyword_ids` (set), or `category_id`/`category_ids`. Optional `difficulty: 'easy'|'medium'|'hard'`. |
| `attempt` | POST | Record answer (or `dont_know`), update `mcat_student_keyword_states` via EMA, return `keyword_states` with `needs_lesson`. |
| `similar` | POST `{session_id, question_id}` | Generate a same-difficulty variant testing the same concept; insert with `parent_question_id`. |
| `quiz` | POST `{session_id, category_id, count?, difficulty?, mixed?}` | Assemble N questions (stored + generated). `mixed: true` spreads easy→medium→hard. |
| `flashcards` | POST `{session_id, category_id, count?, keyword_id?, keyword_ids?}` | Stored + generated flashcards, weakness-ranked. |
| `flashcard-attempt` | POST | Record got_it/missed_it/dont_know, update keyword states. |
| `lesson/[keywordId]` | GET | Cache-or-generate a 3–4 step micro-lesson (stored in `mcat_lessons`). |
| `feedback` | POST `{session_id, content_type, content_id, rating?, flagged?, flag_reason?}` | Rate/flag a question/flashcard/lesson. ≥2 flags → question/flashcard `status='flagged'` (stops serving); lesson deleted (forces regen). |

All routes use the service-role Supabase client and the `{ error, detail }` + 502 contract for generation failures (same as `learnGenerator.ts`).

## Practice loop (`/mcat/[categoryId]/practice`)

**Lessons are the LAST resort, not the opener.** Questions/flashcards are the diagnostic; a lesson only appears on demonstrated struggle or an explicit click.

Per-keyword flow:
1. **New keyword** (`total_attempts === 0 && score === null`) → **flashcard warm-up**: up to 2 keyword-targeted cards ("Quick warm-up"), graded, then questions.
2. Otherwise → straight to **questions**.
3. **Mastery-gated progression** (replaces a fixed block size): keep serving questions on the topic — biasing toward `similar`-question variants when the last answer was correct (~50%) so variants of one concept batch together — until **mastered** (`state === 'mastered'` OR client `topicCorrectStreak >= 3`) or a hard cap of **8** questions (un-mastered → re-queued once at the end). A header "mastery meter" (●●○) shows streak toward 3.
4. **Spaced review between topics**: on advancing, ~35% chance (if `review_pool` non-empty) to insert one review question from a previously-learned topic (prefers `spaced_review_due_at <= now`, else lowest score), badged "Review", before the next topic starts.
5. **Lessons** appear only via the amber struggle banner (2 consecutive wrong / dont_know / server `needs_lesson`, once per keyword) or the always-visible "Learn this" header button. After a lesson, return to the topic's questions (streak reset to 0).

**Difficulty selector** in the header — `Adaptive · Easy · Medium · Hard`, persisted to `localStorage["mcat_practice_difficulty"]`. Non-adaptive modes send `difficulty: <tier>` on every `next-question`.

## Intro keywords — framing-only (`order_index = -1`)

Each umbrella opens with an **intro keyword** (seeded with `order_index = -1`; ids usually end `_intro`, some `_overview` — **always key on `order_index === -1`, never the id string**). It is **`tier='in_depth'`** but is **purely framing**, not a teachable subtopic. The contract (2026-06-26):

- **Lesson** = motivation + why-it-matters + a brief look-ahead that *names* the upcoming subtopics **without teaching or drilling them**. Driven by `opts.isIntro` threaded into `generateMcatLesson` from the lesson route (`kwRow.order_index === -1`). The old id-regex detection and the hardcoded amino-acid "define+preview" carve-out were **deleted** — all intros now share this one framing directive.
- **Flashcards** = general cross-cutting terms only (the umbrella's foundational vocabulary / an "X vs Y" distinction the later subtopics assume), **may be empty**, no complete-enumeration, no empty-retry. `generateMcatFlashcards({ intro: true })`; the `flashcards` route routes intro-scoped generation via its `introIds` set and excludes intros from batch enumeration.
- **Questions — NEVER.** `loadTargetKeywords(supabase, cats, { excludeIntros: true })` drops `order_index === -1` from the `next-question` and `quiz` candidate pools (hard structural guarantee). Flashcard/lesson callers leave `excludeIntros` off so intros still get their lesson + framing cards.
- **Auto flow** (`/mcat/auto`): `auto-plan` returns `intro_ids`; the page shows LESSON → FLASHCARDS then `markSkillMastered` + `advanceKeyword({ wasMastered: false })` to jump to the first real subtopic — it never calls `loadQuestion` for an intro. `beginSkillIntro` always re-runs an intro's warm-up (it never short-circuits to practice), so a missed master self-heals.

## Generation (`apps/student/lib/mcatGenerator.ts`)

Model **`gpt-5.4-mini`**, JSON mode, no streaming, typed `McatGenError` → 502 (mirrors `learnGenerator.ts`). Exports: `generateMcatQuestions`, `generateSimilarQuestion`, `generateMcatFlashcards`, `generateMcatLesson`.

**Two grounding inputs injected into every generation user prompt:**
1. **Template cards** — `fetchTemplateCards()` (`lib/mcatTemplateCards.ts`) pulls real facts from the MileDown anki deck in `anki_cards`, matched by the category's `CATEGORY_TO_TAG_PREFIXES` tag prefixes + term overlap.
2. **AAMC content outline** — `outlineContextForCategory(categoryId)` (`lib/mcatContentOutline.ts`) returns the official AAMC content-category (1A–3B) scope + canonical topic list, so generated content stays in the scope/depth/terminology the real MCAT tests. Injected above the template-card block.

**Reason-first generation + code-assigned answer index** (questions & similar) — to stop wrong answer keys (the model committing to an answer before reasoning, e.g. an explanation concluding "−1" while `correct_index` marks "−2"): the requested JSON order is `stem → explanation (worked solution) → choices`, and `correct_index` is **not model-chosen**. The code picks a random `target_index` (0–3) per question, passes it as an ANSWER PLACEMENT directive, and sets `correct_index` from it after validation — so the key can't drift from the model's reasoning, and the index is uniformly random by construction (no index-0 bias, so `shuffleChoices` is no longer applied to questions; it's still used for **lesson check questions** via `shuffleLessonStep`). Validation requires exactly 4 **distinct** choices.

## Content correctness — fast verification (`mcatGenerator.ts`)

Generation is cheap/creative; a **separate, fast verification pass gates what gets stored/served**. It runs inline (students wait during generation), overlapped with the embedding step, and **fails open** (a timeout/error never blocks serving). Only *newly generated* content is verified — cached/stored content is served as-is.

- **Questions** (`verifyQuestionFast` / `verifyQuestionsFast`): one `gpt-5.4-mini` **blind solve** — the verifier independently answers the question *without seeing the key* and must agree with `correct_index`. Wired into `next-question`, `quiz`, `similar`: only questions that pass are stored/served; if all in a batch fail, serve best-effort (fail-safe). Catches the fatal "wrong answer key" error (~1s, AbortController timeout ~4s).
- **Lessons** (lesson route): every micro-step **check question** is blind-solve-verified before the lesson is cached; if any fail, the lesson is regenerated once and the version with fewer failures is kept (fail-open).
- **Flashcards** (`verifyFlashcardFast` / `verifyFlashcardsFast`): a fast fact-check that the **back correctly answers the front**; only valid cards are inserted (fail-safe to keeping all if every card is rejected).

This is the *correctness* analog of the blueprint (which governs *scope*). NOT yet built: a thorough async strong-model pass (multi-vote + adversarial critic) for offline pool re-checks.

## Concept blueprints — per-keyword scope contracts (`lib/mcatBlueprint.ts`)

Lessons and questions for a keyword are independent LLM calls; without a shared contract they drift (e.g. a sign-of-ΔG keyword getting questions that test the ΔG = ΔH − TΔS formula). The **`ConceptBlueprint`** (stored JSONB on `mcat_keywords.concept_blueprint`) is that contract: `in_scope_concepts`, `in_scope_formulas` (empty for conceptual keywords), `out_of_scope`, `key_terms`, `boundary_statement`.

- **Generated once per keyword** by `generateConceptBlueprint`, **sibling-grounded** (the keyword's siblings are passed in so `out_of_scope` sharply names adjacent territory) + AAMC outline. Encodes a **primary-skill vs shared-outcome** distinction: `out_of_scope` lists concepts that are a *different tested skill* owned by a sibling, NOT shared conclusion vocabulary (e.g. "spontaneous" is a shared outcome, not exclusive to one keyword).
- **Injected** into all four generators via `buildBlueprintBlock` (a forceful "SCOPE CONTRACT" block) — so lessons and questions obey one contract. Routes load `concept_blueprint` (via `loadTargetKeywords` / direct selects) and thread it through.
- **Audit** (`mcat:audit-scope`): an LLM judges each stored question against its keyword's blueprint — flags only when an out-of-scope concept is the *primary tested skill* (not an incidental mention); `--apply` quarantines violators (`status='out_of_scope'`).

## AAMC yield — topic importance (`yield_level` / `yield_rationale`)

Generated in the **same call** as the blueprint (free): `yield_level` ∈ {high, medium, low} + a one-sentence `yield_rationale`, grounded in the AAMC outline (high = explicitly elaborated / foundational / recurring; low = niche). Calibrated to ~25–35% high / 40–50% medium / 20–30% low (not everything "high"). Coerce-with-default ("medium") so it never fails a blueprint. Stored on `mcat_keywords.yield_level` / `yield_rationale`.
- **Surfaced**: `taxonomy` route returns `yield_level` per keyword + an aggregated level per umbrella; `YieldBadge` renders High/Med/Low in the browse + progress pages.
- **Prioritized**: a bounded yield nudge (`high −0.12 / med 0 / low +0.10` on the effective weakness score) in `practice-queue`, `next-question`, and `quiz` — high-yield topics surface earlier without overriding weakness; an explicitly pinned `keyword_id`/`keyword_ids` is never reordered.

## Difficulty model

`difficulty` on `mcat_questions` is a continuous **[0.2, 0.9]** value, **LLM-set within a requested band** (so "hard" is genuinely hard — see [difficulty-scales.md](difficulty-scales.md) for the band definitions and how this differs from precalc's 1–5 / `estimated_difficulty` scales).

- **Generation:** `generateMcatQuestions` takes `targetDifficulty?: number` and/or `difficultyTier?: 'easy'|'medium'|'hard'`; the prompt describes EASY (single-step recall), MEDIUM (one inference), HARD (multi-step reasoning / two concepts / all-plausible distractors). `generateSimilarQuestion` inherits the source question's difficulty.
- **Adaptive target** (`next-question`): `base = 0.2 + avgStrength*0.6`, plus an escalation bump `min(0.15, 0.05 * maxConsecutiveCorrect)` over the scoped keywords, clamped to [0.2, 0.9]; the tier is derived from the target and both are passed to generation. An explicit `difficulty` body param overrides the adaptive target and biases stored-candidate selection toward that band (2× in-band multiplier on the difficulty-fit gaussian).
- **Candidate scoring** (stored questions): `weakness × difficultyFit(gaussian σ=0.2) × ratingNudge(0.7 + 0.3*(avg_rating/5))`, weighted-random pick from top 5.

## Mastery state & EMA (`mcat_student_keyword_states`)

One row per `(session_id, keyword_id)`. **Kept normalized (not a JSONB blob)** because every read needs per-keyword querying: weakest-N selection, umbrella implied-score averaging, `spaced_review_due_at` filtering, and atomic per-keyword upserts under concurrent attempts. Same rationale and shape as the precalc `learn_student_keyword_states`.

Per attempt, for each keyword in the question's `keyword_weights`:
- EMA via `updateStrengths` (`lib/practiceAlgorithm.ts`, α=0.12, weight-scaled, first-touch target 0.75/0.25). `dont_know` counts as incorrect and increments `dont_know_count`.
- `state = 'mastered'` when `score >= 0.8 && consecutive_correct >= 4`; on mastery it schedules `spaced_review_due_at` (via `computeNextReviewDate`, REVIEW_INTERVALS `[1,3,7,14,30,60]`) and bumps `spaced_review_count`. `needs_lesson` when `score < 0.35 && total_attempts >= 3`, or immediately on a `dont_know`. Else `in_progress`.
- Keyword ids not present in `mcat_keywords` are skipped (FK safety).

## Embedding & keyword tagging

`text-embedding-3-small`, stored as JSONB arrays. `npm run mcat:embed` runs three phases: keywords (`label. description`), questions (`stem | correct_choice`), flashcards (`front | back`). Questions and flashcards are **retagged** after embedding: cosine similarity vs the in_depth keywords of their category → top 4 with sim > 0.25 (fallback top 2), normalized to sum 1. `cosineSimilarity` + `tagByEmbedding` live in `lib/mcatTagging.ts`. New content is embedded+retagged inline on generation (try/catch — embedding failure never fails generation; the row is left for the batch script).

`npm run mcat:clean-unembedded` deletes any question/flashcard still missing an embedding (deletes dependent attempts first for FK safety) — guarded behind `--apply` (dry-run by default).

| Command | Script | Purpose |
|---|---|---|
| `npm run seed:mcat` | `seed-mcat-taxonomy.ts` | Seed `mcat_categories` + `mcat_keywords` from `mcat-keywords.txt`. `--dry-run`. |
| `npm run mcat:expand` | `expand-mcat-keywords.ts` | Generate 5–8 in_depth keywords per childless umbrella. `--dry-run`, `--limit N`, `--category <id>`. Idempotent (skips umbrellas that already have children). |

| `npm run mcat:blueprints` | `backfill-mcat-blueprints.ts` | Generate + store `concept_blueprint` **and** `yield_level`/`yield_rationale` per keyword (one LLM call each). **Fill-missing** by default (full blueprint where absent, yield-only where the blueprint exists but yield is null); `--force` re-generates. | `npm run mcat:audit-scope` | `audit-mcat-scope.ts` | LLM-audit stored questions vs their keyword's blueprint (primary-skill, not incidental mentions). Report-only by default; `--apply` quarantines violators (`status='out_of_scope'`). Scope with `--umbrella`/`--category`/`--keyword`. |
| `npm run mcat:rebalance` | `rebalance-mcat-correct-index.ts` | One-time/legacy: shuffle stored questions' choices to fix index-0 bias (superseded for new questions by code-assigned random `target_index`). `--dry-run`. |

| (no npm alias) | `test-blueprint-scope.ts` | Read-only harness: generates a keyword's blueprint + yield in-memory and measures question scope-drift with vs without it. `--keyword`/`--count`/`--rounds`. |
