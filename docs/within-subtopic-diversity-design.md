# Within-Subtopic Question Diversity — Design

> **v2.** Adds the **batch-generation-on-cache-miss "recycle model"** as a first-class strategy (§4) — the unit of generation becomes a small, mutually-distinct *batch*, of which one is served immediately and the rest backfill the pool to become the next served questions, recycled for all future users. v2 also reframes the diversity menu as a per-subtopic **whitelist** (most cells are correctly empty), treats target difficulty as a soft *input* while the post-hoc measured 0–1 stays the ground truth, and clarifies that batch (deep+diverse pool) and MMR/recycling (smart serving) are complementary.

**Executive summary.** Within a single subtopic, Lodera's practice questions are too similar to each other — a student grinding "alpha helix" or "evaluate a limit by factoring" sees the same task, the same representation, and the same one misconception over and over. This is not a content-depth problem; it is a *structural* one. Each question is generated independently from the same inputs (subtopic + its description + a fixed house-style exemplar block + a target difficulty), and the only thing that ever de-duplicates two items is a token-level Jaccard comparison of their **stem text** (`questionDiversity.ts` `NEAR_DUP_THRESHOLD = 0.55`). Selection then re-clusters because it scores purely on weakness × difficulty-fit and never looks at *what kind* of question an item is. Meanwhile the system already computes the exact signals that describe a question's "kind" — an **action description**, a **representation description**, **per-distractor misconception descriptions**, and now a **prerequisite description**, each embedded (`questionEnrichment.ts`) — but it uses them only as after-the-fact tags for analytics, never as inputs to generation or selection. The core recommendation is to promote those signals from *outputs* to *inputs*: plan a spread of (archetype × representation × solution-schema × targeted-misconception × difficulty) **before** generating, and diversify serving on the **action/representation embeddings** rather than on stem text. Because latency is the #1 retention blocker, the recommended first step is entirely on the cheap, no-new-LLM-call path: load the embeddings selection already ignores and add a max-marginal-relevance / coverage-quota pass in the existing `next-question` and `practice-queue` routes. The second pillar is the **recycle model** (§4): a cache-miss already generates a *batch* (`count: 3` today) and stores all passing items (`next-question/route.ts:463,510–538`), but serves only one — so the extras are latent inventory. Make that batch deliberately diverse and deep enough, and almost every future request is a cheap recycle of a pre-vetted, materially-different stored item instead of a slow live generation. Generation is the expensive, slow step; recycling is free. The two pillars compose: the batch makes the pool deep and diverse; MMR makes recycling smart (serve the most-different next item) without any hot-path LLM call.

## 1. Problem statement and root cause

The symptom is documented directly in the judge reports: 47–57% of a 32-question MCAT "Amino Acids and Proteins" session were alpha-helix items, and 5 of 7 "polar uncharged" questions were "effectively the same question" (`judge-mcat/REPORT.md`). The math judge saw the same shape at the subtopic level — repeated near-identical "read the limit notation" stems. The streak guards we shipped (below) reduce *keyword*-level repetition but do nothing about *within-keyword* sameness, which is the real complaint.

There are three independent code paths that each contribute, and it helps to separate them.

**Generation produces clones because every call gets the same inputs.** A single question is produced by `generateMathQuestions` (`apps/student/lib/mathGenerator.ts:546`) / `generateMcatQuestions` (`apps/student/lib/mcatGenerator.ts:393`). The user prompt is literally `Generate ${count} AP math multiple-choice question(s)` (`mathGenerator.ts:576`) plus a keyword block (id + label + `description` + blueprint scope) and a *fixed* house-style exemplar block (`exemplarSection`, `mathGenerator.ts:572`, sourced from `mathExemplars.ts`). The only diversity instruction anywhere is "EXACTLY 3 plausible-but-wrong answers, each embodying one specific predictable student error, all distinct from one another" (`mathGenerator.ts:276`) — i.e. *distractors within one item* must differ, but nothing tells the model to make item B differ from item A, or from anything the student already saw. When the model is asked for the same subtopic with the same description, the same exemplars, and the same target difficulty, it regresses to the most prototypical question for that subtopic. Difficulty is chosen "after" and then clamped/nudged into a band (`clampDifficulty`, `mathGenerator.ts:95`), so even difficulty doesn't spread items apart within a tier.

**Enrichment knows the difference between two items but isn't consulted at generation.** After a question is inserted, `enrichQuestionsInBackground` fires via Next's `after()` (`questionEnrichment.ts:228`) and `enrichOne` (`questionEnrichment.ts:156`) generates and embeds five descriptions — `problem_description`, `action_description`, `representation_description`, `prerequisite_description`, and a per-wrong-choice `wrong_answer_descriptions[]` — then tags them via cosine (`nearestWeights`, `questionEnrichment.ts:55`) against `math_action_keywords`, `math_representation_keywords`, and the in-depth `math_keywords` content pool. These are exactly the coordinates that say "this is a *graph-reading* item testing the *one-sided-limit* misconception" vs "this is a *symbolic factor-and-cancel* item." But this runs *after* insert and only writes tag columns; it is never read back to influence the next generation.

**Selection re-clusters because it scores on weakness and difficulty only, and dedups on stem text only.** In `apps/student/app/api/math/next-question/route.ts` the candidate query selects `id, stem_latex, choices, correct_index, solution_latex, hint_latex, keyword_weights, difficulty, parent_question_id, avg_rating` (`:272`) — note the absence of any `*_embedding` or description column. The score is `weakness × difficultyFit × ratingNudge × inBandBoost × prioBoost` (`:300–320`), where `difficultyFit` is a Gaussian on the 0–1 difficulty (`:96`) and `weakness` is the weighted `(1 - strength)` over the item's keywords (`:306`). The only diversity applied is the streak guards plus `filterNearDuplicates(candidates, seenStems)` (`:340–343`), which calls `jaccardSimilarity` on `normalizeStem` output (`questionDiversity.ts:42,65,93`). Two questions that are conceptually identical but worded differently — "find $\lim_{x\to2}\frac{x^2-4}{x-2}$" vs "evaluate $\lim_{x\to3}\frac{x^2-9}{x-3}$" — share few tokens after `NUM`-collapsing, so Jaccard stays well below 0.55 and both are served. The MCAT route (`apps/student/app/api/mcat/next-question/route.ts`) is identical in shape. The batch builders `practice-queue/route.ts` (math and mcat) apply **no** diversity at all — they order purely by weakness + a yield nudge (`practice-queue/route.ts:304–312`) and return the top 40.

In short: independent generation from identical inputs creates near-clones; stem-token dedup is the wrong distance metric for "same kind of question"; and selection has no notion of "kind" to spread on.

## 2. The core insight: turn the tag signals into the control signals

The system already manufactures, per question, the four coordinates that define within-subtopic variety:

- **Action** — the cognitive task ("factor, then substitute"; "read a value off a graph"). Stored as `action_description` + `action_description_embedding` + `action_weights`, tagged to the `math_action_keywords` pool (`questionEnrichment.ts:191,205–219`).
- **Representation** — graph / table / symbolic expression / equation / word problem. Stored as `representation_description` + `representation_description_embedding` + `representation_weights`, tagged to `math_representation_keywords`.
- **Targeted misconception(s)** — one named student error per distractor. Stored in `wrong_answer_data` as `{description, embedding, keyword_weights}[]`.
- **Prerequisite** — the assumed sub-skills, in general terms. Stored as `prerequisite_description` + `prerequisite_description_embedding`.

Today these are *descriptions of what the model happened to produce*. The design principle of this doc is to **invert the data flow**: choose the action, representation, schema, and targeted misconception *first*, hand them to the generator as a contract, and *select* on the action/representation embeddings so the served sequence walks across the space instead of sitting in one corner of it. The enrichment pipeline barely changes — it becomes a *verifier* that the realized item matches its plan, rather than the only place these signals exist.

## 3. Generation-time diversity

### 3.1 Two-stage "problem plan → realize"

Replace "ask for N questions on a subtopic" with two cheaper, more controllable steps.

**Stage A — plan.** Produce a set of lightweight *plans*, each a small struct:

```
{ archetype, representation, schema, targeted_misconception, difficulty_band, edge_case? }
```

A plan is ~30 tokens, contains no math, and is trivial to dedup. Generate a pool of plans (or draw them from a per-subtopic taxonomy — §6), then **spread** them: drop any plan whose `(archetype, representation, schema)` triple is already chosen, and—if you want semantic spread without authoring a taxonomy—embed the plan strings and keep the most mutually dissimilar K (farthest-point / k-center sampling). This is the same dedup idea as `filterNearDuplicates`, but applied to *plans* on the diversity axes that matter, not to realized stem text.

**Stage B — realize.** For each surviving plan, call the existing generator with the plan injected into the prompt as a hard contract: "Task archetype: …. Representation: …. Use solution method: …. The single most attractive distractor must encode this misconception: …. Difficulty band: …." This slots directly into `mathGenerator.ts`'s `userPrompt` (`:576`) and `mcatGenerator.ts` (`:653`) — it is an additional grounding block alongside the keyword and exemplar blocks, not a rewrite. The realized item is then enriched exactly as today (`enrichOne`), and enrichment doubles as a check: if the realized `action_description_embedding` is far from the plan's intended archetype, flag/regenerate.

The axes to spread on:

- **Task archetype → action keywords.** The verb of the problem. For limits: *evaluate-by-direct-substitution*, *evaluate-by-factoring*, *evaluate-by-conjugate*, *one-sided-limit-from-a-graph*, *infinite-limit/asymptote*, *limit-of-a-piecewise-at-a-junction*, *determine-whether-a-limit-exists*. These map onto entries in `math_action_keywords` (the pool `nearestWeights` already tags against), so the taxonomy and the existing tag vocabulary can be the same list.
- **Representation.** graph / table of values / symbolic expression / equation / verbal word problem. Maps onto `math_representation_keywords`. A subtopic that is 90% symbolic today *may* warrant a quota of graph and table items — but only if those representations are in-scope for that subtopic (see the whitelist caveat below and §6); a purely-symbolic factoring subtopic legitimately has no graph or table form, and forcing one would manufacture off-scope items.
- **Solution schema (sub-method).** Within one archetype, the *path* to the answer. Concrete calc example: for "evaluate a limit that is $0/0$," the schemas are *factor-and-cancel* (difference of squares, common factor), *rationalize-the-conjugate*, *expand-then-simplify*, and *apply-a-known-special-limit*. Two factor-and-cancel items are near-clones even with different numbers; a factor item next to a conjugate item is genuinely different practice. Schema is the axis stem-Jaccard is blindest to — **and it is the safest axis to lean on**, because alternate sub-methods are by construction in-scope for the subtopic, whereas swapping representation or archetype risks drifting off-scope. Prefer schema spread first; treat representation/archetype spread as opt-in per the subtopic whitelist.
- **Targeted misconception.** Each plan names the dominant distractor error, drawn from the misconception space the per-distractor `wrong_answer_data` already catalogs — e.g. "plugged in the hole value," "kept the removable factor," "took the one-sided limit as two-sided," "sign error on the conjugate." Forcing the *primary* distractor to vary across items is what makes a cluster feel like it's probing different failure modes rather than re-asking one.
- **Edge cases.** Negatives, fractions/non-integers, zero, parameters instead of constants, junction points, domain restrictions. A cheap, high-value spread axis.

**Difficulty is a soft *input* target, not a label.** A plan carries an intended `difficulty_band` to *aim* the generator, but the spec's contract — difficulty is **measured after** the problem exists — is preserved exactly: the realized item still goes through `clampDifficulty` (`mathGenerator.ts:95`) and the stored 0–1 value is re-measured, never assumed from the plan. Selection continues to use only that stored ground-truth value in `difficultyFit` (`next-question/route.ts:96`). Aiming and measuring coexist: the plan biases the model toward a band so a batch can be spread into a small **difficulty ladder** (§4), but the re-measure is never skipped and the plan's band is never written as the truth.

**The axis menu is a per-subtopic *whitelist*, not a full grid.** Most (archetype × representation) cells are *correctly* empty: "limits from a graph" is inherently graphical and has no symbolic-only form; a binomial-factoring subtopic has no graph/table form. Spreading must therefore target the subtopic's **valid** cells only — never "fill every cell." How the whitelist is sourced is detailed in §6; the generation-time consequence is simply that Stage A draws plans from that subtopic's allowed axis set, and the scope gate (blueprint contract / `mcat:audit-scope`) remains the backstop: **diversity proposes, scope disposes.**

### 3.2 Cheaper prompt-level levers (no new architecture)

Even without the plan/realize split, four prompt-level changes materially help and cost nothing or one extra generation:

1. **Pass the last-N stems with an explicit "make it materially different."** The route already receives `recent_keyword_ids` and `seen_stems` from the client (`next-question/route.ts:147–158`); thread `seen_stems` into the generator prompt as "Do NOT reproduce the structure of these recently-seen items; change the representation and the solution method." Zero extra calls.
2. **Diverse few-shot exemplars.** `exemplarSection` (`mathGenerator.ts:572`) is currently a *fixed* house-style block. Rotate it per call to span representations and schemas (one graph exemplar, one factoring exemplar, one conjugate exemplar) so the model's prior isn't anchored to one prototype.
3. **Batch-generate-and-keep-most-dissimilar.** The generator already returns an array of `count` items; over-generate (ask for more than needed) and keep the subset that is most mutually dissimilar on the action/representation embeddings. One generation call, plus embeddings (cheap), no extra LLM round-trip on the hot path if done in the background-refill path.
4. **Explicit per-item diversity instruction.** Add to the system prompt that across the requested items, archetype and representation must each vary — a one-line change to `QUESTION_SYSTEM` (`mathGenerator.ts:257`, `mcatGenerator.ts:225`).

Levers 1, 2, and 4 add **no** latency; lever 3 adds only embedding cost and only when generating.

## 4. Batch generation on cache-miss (the recycle model)

This is the strategy that turns diversity work from a per-request tax into a one-time, amortized investment, and it is the natural home for everything in §3. It is also already half-built.

**Trigger — a cache-miss.** The live path falls into generation in exactly two situations, and they deserve different treatment. The first is **"none exist"** (cold): there is no stored question for the requested `(subtopic × difficulty band)` cell at all — true on a freshly seeded catalog, or for a subtopic no one has reached yet. The second is **"seen all, but others exist"** (warm): stored questions exist for the cell but the student has already seen every one that survives the diversity filters (`recent_keyword_ids`/`seen_stems` in `next-question/route.ts:147–158` plus the streak/near-dup guards). Today both fall through the stored-pool branch (`:354–382`) into the same generation block (`:451–543`). The cold case warrants a larger **bootstrap** batch; the warm case warrants only a couple of fresh items to top the cell back up.

**Mechanism — generate a batch, serve one, recycle the rest.** The crucial reframing the product owner emphasized: the extra items are *not* thrown away and are *not* waited on. On a miss, generate a batch (say 1 + 4, larger when the catalog is cold), serve **one** immediately, and let the other four be **stored and consumed as the user's subsequent questions** — and as future questions for *every* user who reaches that cell. This is not speculative: the code already does the storage half of it. `generateMathQuestions({ ... count: 3 })` (`next-question/route.ts:463`) produces a batch, `verifyMathQuestionsFast` filters it (`:516`), and **all passing items are inserted** with `source: "generated", status: "active"` (`:510–538`), then one is served and the rest are enriched in the background via `enrichQuestionsInBackground` (`:547`). What is missing is intent: the extras are currently a side effect, not deliberately sized, not deliberately diverse, and not difficulty-laddered. The recycle model makes them the point.

**Economics — first N users pay, everyone after recycles.** Generation (the LLM call + verify + embed + tag) is the expensive and slow part; reading a stored row is effectively free. If each miss deposits four vetted, mutually-distinct items, then four of the next several requests for that cell are cheap stored serves rather than slow generations. Over a cohort, the per-question generation cost asymptotes toward zero: the first handful of users to touch a cell pay for it, and the long tail recycles. A cache-miss is the *correct* trigger precisely because it fires exactly when (and only when) inventory is short.

**Why the batch is the ideal unit of diversity.** Forcing five items *generated together* to be mutually distinct is both cheaper and more reliable than trying to make independently-generated singletons differ after the fact. It pairs exactly with §3.1: generate a diverse **set of plans** (Stage A), then realize the **batch** (Stage B) — one planning pass amortized over the whole batch. It also pairs with difficulty-as-input (§3.1): spread the batch into a small **difficulty ladder** — e.g. target−1, target, target, target+1 — so the "harder on correct / easier on wrong" escalation (`escalationBump`, `next-question/route.ts:262`) has inventory ready at the adjacent bands instead of having to generate again the moment the student gets one right. A batch keyed to a `(subtopic × band)` cell but spread across representation/archetype/schema and a difficulty rung is the unit that makes recycling worth doing.

**One call for five vs five calls with anti-similarity context — a real tradeoff.** Asking for all five in a single generation call is cheaper and lets the model see its own siblings (so it can self-differentiate), but risks *correlated* variation — five items that differ only cosmetically, or a shared subtle error. Five separate calls, each handed the previously-generated stems with "make this materially different," are more independent and harder to make trivially-similar, but cost ~5× the calls and latency. For the **served** item latency is bounded either way (you return the first as soon as it is ready); for the **recycled** items, the single-call route is the pragmatic default, with per-call anti-similarity reserved for cold bootstrap batches where pool quality matters most.

**Where it can run.** Because the four extras are simply consumed as the *next* served questions, the simplest implementation needs no separate worker at all: the existing same-request path already returns one and persists the rest (`:536–547`). For the larger cold bootstrap batches, a background job/queue is cleaner — it keeps the cold-start request from blocking on a big generation and lets enrichment (which already runs via `after()`, `questionEnrichment.ts:228`) complete before the extras are eligible to serve.

### Critical risks and what they demand

- **The extras must pass the *full* pipeline, offline, before they are servable.** A recycled item that skipped solve-verify, scope-check, embed, or tag is worse than no item — it pollutes the pool permanently and is then served to everyone. The batch path must run `verifyMathQuestionsFast` (it does, `:516`) **and** complete enrichment + the blueprint/scope gate **and** an intra-batch dedup before an extra is marked eligible. Practically this argues for a servable flag: insert extras as `status: 'pending'`, flip to `'active'` only after enrichment + scope pass, so a half-processed extra can never be served. (Today everything is inserted `active` immediately — fine for a served item that was just verified, riskier for unseen extras.)
- **Cold-catalog cost spike.** Early on, *every* request misses, so every request spawns a batch — a thundering-herd of generation right when the catalog is emptiest. This needs (1) a **budget/concurrency cap** on concurrent batch generations, and (2) an **in-flight CLAIM/LOCK per cell** so two users hitting the same empty `(subtopic × band)` cell do not each spawn a full batch — the second should wait for or be served by the first's output. Without the lock, a popular cold cell gets N redundant batches.
- **Catalog shape vs recycle depth.** Key the pool **coarsely** — `(subtopic × difficulty band)`, not by exact archetype×representation×schema — and force diversity *within* the batch across those finer axes. Keying finely explodes the number of cells and defeats recycling (every fine cell is perpetually cold). Per-cell depth must be sized to **real usage**: a heavy user doing ~15–20 items on a subtopic across spaced reviews needs a pool deeper than that, or recycling just re-triggers a miss every session. Depth target should track measured per-subtopic consumption, not a fixed constant.
- **Two triggers, two batch sizes.** Conflating them wastes money or under-stocks. "None exist" → a larger bootstrap batch (seed the cell's valid cells per the §6 whitelist). "Seen all but others exist" → a couple of fresh items only.
- **Bounded waste.** Some pre-generated extras are never seen — the student masters or abandons the subtopic. This is cheap, bounded, one-time waste (a few items per dropped cell) and is the acceptable cost of the amortization; it should be *measured* (unserved-generated ratio) but not over-engineered away.

## 5. Selection-time diversity

This is where the biggest, cheapest win lives, because the signals already exist in the DB and selection simply isn't reading them.

**Step 0 — read the embeddings selection currently drops.** Add `action_description_embedding, representation_description_embedding, problem_description_embedding` (and optionally `prerequisite_description_embedding`) to the candidate `.select(...)` at `next-question/route.ts:272` and the MCAT equivalent. No new computation — these columns are already populated by enrichment.

**Min-distance / max-marginal-relevance serving.** Keep the current relevance score (`weakness × difficultyFit × …`, `:300–320`) but, when choosing among the top-K (the existing weighted-random over the top-5 at `:354–382`), penalize candidates that are close in **action/representation embedding space** to items in `seen` this session. MMR form: `final = relevance − λ · max_cosine_to_recent(action⊕representation embedding)`. This replaces "pick the highest-weakness item" with "pick the most-useful item that is also *different in kind* from what you just did," and unlike `filterNearDuplicates` it measures conceptual distance, not token overlap.

**Cluster-and-round-robin.** Within a category, cluster the candidate pool by archetype (k-means or simple nearest-action-keyword bucket, since `action_weights` already names the bucket) and serve round-robin across clusters. Cheap to implement on top of `action_weights` without even needing the raw embedding.

**Diversify on action/representation, not problem text.** The crucial change from today: the distance that drives spreading should be the action⊕representation embedding, because two items can share almost no stem tokens (passing Jaccard) yet be the identical task. `filterNearDuplicates` stays as a cheap backstop for literal repeats; the embedding MMR is the real diversity engine.

**Per-subtopic coverage quotas over the subtopic's *valid* cells.** In the batch builders (`math/practice-queue/route.ts`, which today applies no diversity, `:304–312`), build the queue to *cover the cells that are in-scope for the subtopic* (per the §6 whitelist), not an abstract full grid: when a subtopic legitimately has, say, four schemas and two representations, fill the queue so it samples across those before repeating any cell, subject to the weakness ordering as a tiebreak. This is the queue analog of round-robin and is where spaced/interleaved review (the `review_pool`, `practice-queue/route.ts:315`) can be made to pull *different* facets of a past subtopic rather than the same one. The metric is spread across *valid* cells — never "fill every cell" (§6).

Each of these reads only columns that already exist; none adds an LLM call.

**Complementarity with the recycle model.** Selection diversity and the §4 recycle model are two halves of one mechanism, not alternatives. The batch model makes the stored pool **deep and diverse** — without it, MMR is choosing among near-clones and can only pick the "least bad" repeat. MMR/round-robin makes recycling **smart** — given a deep, diverse pool it serves the most-different next item, which is exactly what makes a recycled (never freshly generated) sequence feel varied. Critically, MMR/recycling adds **no hot-path LLM call** (it is cosine math over already-stored rows), and the batch is what makes the recycled pool worth serving from. Ship selection diversity first (it is cheap and immediately useful even on today's shallow pool), then let the batch model deepen the pool underneath it.

## 6. Taxonomy and measurement

**The diversity menu is a per-subtopic *whitelist*, and most cells are correctly empty.** The single most important taxonomy decision is that each subtopic gets its *own* allowed set of axes — not a shared global grid. Many (archetype × representation) combinations are off-scope for a given subtopic: a "limits from graphs" subtopic is inherently graphical and has no symbolic-only form; a "factor a binomial with leading coefficient 1" subtopic has no graph or table representation at all. The target metric must therefore be **spread across the subtopic's valid cells**, never "fill every cell" — a full-grid quota would actively manufacture off-scope, low-quality items, the opposite of the goal.

**Sourcing the whitelist.** Build it from two reinforcing signals, both offline: (a) **LLM-propose** the valid axis set per subtopic once, from the keyword `description` + examples *already stored* on `math_keywords` (and the `concept_blueprint`) — ask the model "which task archetypes, representations, and solution sub-methods are legitimately in scope for this subtopic?" and store the reviewed result alongside the taxonomy in `content/math-taxonomy/*.json`; and (b) **reinforce from reality** — the realized `action_weights` / `representation_weights` tags of questions that have *already passed the scope gate* are empirical evidence of which cells are valid, so the whitelist can be widened or pruned by what scope-approved content actually occupies. The action and representation *keyword* pools (`math_action_keywords`, `math_representation_keywords`) are the controlled vocabulary these map onto, so generation, tagging, and quotas all speak the same language.

**Lean on schema; keep scope as the backstop.** Of the axes, the **solution-schema** axis (alternate in-scope sub-methods) is the safest to drive spread, because every schema is by construction within the subtopic; representation and archetype spread are higher-variance and must be gated by the whitelist. And whatever diversity proposes, the existing **scope gate** disposes: the blueprint contract / `mcat:audit-scope` audit remains the final filter on any generated item. Diversity is a proposal mechanism; scope is the veto.

**Difficulty: aim with an input, trust the measurement.** The taxonomy carries a target difficulty band per plan as a generation *aim*, but the stored 0–1 difficulty is always the post-hoc **measured** value (`clampDifficulty`, `mathGenerator.ts:95`) and is the only difficulty selection ever reads (`difficultyFit`, `next-question/route.ts:96`). This reconciles with the spec's "difficulty is measured after the problem is created": aiming is allowed, but the re-measure is never skipped and the aim is never written as truth. Keep both signals; never collapse them.

**Metrics the judges can auto-flag.** Define a per-subtopic diversity score computed offline over the stored pool (and per served session):

- **Pairwise within-subtopic cosine** on the `problem_description_embedding` (and separately on action/representation embeddings) — mean and max. A subtopic whose mean pairwise cosine exceeds a threshold (e.g. > 0.85) is "too samey" and flagged for regeneration. This is the quantitative version of the judge's "5 of 7 were the same question."
- **Archetype entropy over *valid* cells** — Shannon entropy of the distribution of items across the subtopic's *whitelisted* action-keyword buckets (normalized by the count of valid buckets, not all buckets); low normalized entropy = one in-scope task dominates. A subtopic with only one valid archetype is *not* penalized.
- **Representation distribution vs the whitelist** — share of items per representation bucket, compared against the subtopic's *allowed* representations. Flag a subtopic only when it under-covers a representation that is whitelisted-valid (e.g. a "limits from graphs and tables" subtopic that is 95% symbolic) — never flag a legitimately single-representation subtopic for being 100% that representation.
- **Session-level repetition** — for a delivered sequence, the max run length within an archetype and the fraction of consecutive pairs above a cosine threshold. This is what the judge harness should assert on directly (it already counts per-topic concentration manually).

These metrics are also the **validation instrument** for any rollout step: ship a change, recompute, confirm mean pairwise cosine drops and archetype entropy rises without weakness-targeting accuracy regressing.

## 7. Phased rollout — impact vs effort vs latency

Latency is the product's #1 retention blocker (`docs/retention-qa-report.md`), so each phase is annotated with whether it touches the hot path.

**Phase 0 — measurement (no latency, no user-visible change).** Implement the §6 metrics as an offline script over the stored pools and wire the two assertions (max within-archetype run length, pairwise-cosine ceiling over *valid* cells) into the judge harness. This is prerequisite to validating everything else and is pure read-only analytics. *Effort: low. Impact: enabling.*

**Phase 1 — selection diversity on existing embeddings (RECOMMENDED FIRST STEP; no new LLM call).** Add the `*_embedding` columns to the candidate `.select` (`next-question/route.ts:272` + MCAT), and add an MMR penalty / cluster-round-robin over action⊕representation embeddings to the existing top-K choice (`:354–382`), plus valid-cell coverage quotas in the `practice-queue` builders. This is the highest impact-per-latency move: it directly breaks the within-subtopic clusters the judges flagged, it reads only already-populated data, and it adds **zero** hot-path LLM calls — only a few cosine computations over an in-memory candidate list. It is immediately useful even on today's shallow pool, and becomes far more useful once Phase 2 deepens that pool. *Effort: low–medium. Impact: high. Latency: none.*

**Phase 2 — batch-on-miss recycle model (§4) — both a latency fix and the diversity vehicle.** Make the existing cache-miss batch deliberate: size it by trigger (bigger bootstrap when the cell is cold/empty, a couple of items when warm), spread it across the subtopic's *valid* axes and a small difficulty ladder, insert extras as `pending` and promote to `active` only after verify + enrichment + scope pass, and add a **per-cell in-flight claim/lock + a global concurrency/budget cap** so a cold catalog can't thunder-herd. The storage half already exists (`next-question/route.ts:463,510–547`); this phase adds intent, gating, and the lock. It *reduces* steady-state latency (more requests become cheap recycles of stored items) at the cost of a bounded cold-start generation spike and some never-served waste. This is the engine that makes recycling — and therefore Phase 1's MMR — worth running. *Effort: medium. Impact: very high. Latency: neutral-to-positive on the hot path; one-time cold spike, capped.*

**Phase 3 — cheap generation levers (no/low latency).** Apply §3.2 levers 1, 2, 4 (pass `seen_stems` + "make it materially different," rotate exemplars across representations, add an explicit cross-item variety instruction). These touch only the generation path that already runs on a miss, sharpen the quality of each batch, and add no extra round-trips. *Effort: low. Impact: medium. Latency: none.*

**Phase 4 — taxonomy whitelist + quota-driven background backfill (latency only offline).** Author or LLM-propose the per-subtopic *valid-axis whitelist* (§6), then run a **background** job that proactively fills each subtopic's empty or over-clustered *valid* cells (over-generate-and-keep-most-dissimilar). This is the offline, scheduled sibling of the on-demand Phase 2 batch: it warms cells before students reach them, so the hot path serves diverse *stored* questions and rarely generates live — which *improves* perceived latency over time. *Effort: medium–high. Impact: high. Latency: none on hot path (improves it).*

**Phase 5 — two-stage plan→realize feeding the batch (defer live use).** Apply the §3.1 plan→realize split as the way Phases 2 and 4 construct each batch (one planning pass amortized over the whole batch). The plan step is an extra LLM call, so it should run **inside the background backfill or the async portion of the batch**, never as a synchronous step in front of the served question. Running it live is the last resort and must be measured against the retention bar. *Effort: medium. Impact: high. Latency: free if kept off the synchronous served-question path; adds one call if not — keep it off.*

### Risks and tradeoffs

- **Diversity vs targeting.** Spreading on archetype can pull the served item away from the student's actual weakness. Keep weakness as the primary score and diversity as a *tiebreaker/penalty* (the MMR λ), not an override — and validate that weakness-targeting accuracy (do struggling keywords still get surfaced?) does not regress.
- **Empty-cell fallback.** Quotas and MMR must degrade gracefully exactly like the existing guards do (`filterStreakKeyword`/`filterNearDuplicates` already fall back to the full pool when filtering empties it, `questionDiversity.ts:108,162,216`). Reuse that pattern.
- **Tag quality.** Selection-time diversity is only as good as the `action_weights`/`representation_weights` tags, which come from cosine top-2 (`nearestWeights`, `questionEnrichment.ts:55`). If the action/representation keyword pools are thin or mis-embedded, clustering is noisy; Phase 0 metrics will expose this.
- **Over-generation cost / bounded waste.** Lever 3 and Phases 2/4 spend more generation tokens, and some recycled extras are never served (the student drops the subtopic). This is acceptable — it is one-time, amortized across many future serves, and bounded per cell — but the unserved-generated ratio should be tracked so the waste stays small.
- **Pool integrity for recycled items.** A recycled item is served to *everyone*, so an extra that skipped any pipeline stage is a permanent pollutant. The `pending`→`active` promotion gate (verify + enrich + scope) is the hard requirement here; never serve an unpromoted extra.
- **Cold-start thundering herd.** Without the per-cell claim/lock and the global concurrency cap, an empty popular cell spawns redundant batches and a cost spike. These caps are not optional for Phase 2.

### What to measure to validate

Per subtopic, before/after: **mean and max pairwise within-subtopic cosine over valid cells** (should fall), **normalized archetype/representation entropy** (should rise toward the valid-cell ceiling, *not* toward a full-grid uniform), and at the session level the **max within-archetype run length** and **fraction of high-cosine consecutive pairs** (should fall). Recycle-model health: **cache-miss rate** and **share of served questions that were stored vs freshly generated** (the recycle ratio — should climb toward ~1 as cells warm), **unserved-generated ratio** (bounded waste), and **concurrent batch generations** (must stay under the cap). Guardrail metrics that must *not* regress: weakness-targeting hit rate (are weak keywords still prioritized?), mastery-progression rate, scope-audit pass rate of generated items, and — above all — **inter-question latency**.
