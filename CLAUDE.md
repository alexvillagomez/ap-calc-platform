## Canonical product spec (authoritative — 2026-06-20)
This is the spec the app must follow. When code and this section disagree, fix the code.

**Problem generation — every problem has 8 PARTS:** (1) PROBLEM — inline LaTeX, short ("Solve…/simplify…"), within the subtopic, requires thinking, NOT memorization (that's flashcards). (2) SOLUTION — clear, shows approach/intuition, skips lines, easy to read. (3) WRONG ANSWERS + WRONG-ANSWER DESCRIPTIONS — 3 plausible (non-random) distractors; each has a description explaining the mistake that produces it; descriptions are GENERAL (embedded). (4) PROBLEM DESCRIPTION — general topics, no numbers (embedded). (5) ACTION DESCRIPTION — what action solves it (embedded, matched to ACTION keyword pool). (6) REPRESENTATION DESCRIPTION — table/graph/expression/equation (embedded). (7) PREREQUISITE DESCRIPTION — prereq knowledge in simple terms (embedded; tagged to general topic keywords). (8) DIFFICULTY — chosen AFTER creation, a NUMBER 0–1, predefined scales; stored in JSON with a pre-determined correct index.

**Keyword tiers:** course (Calculus / Pre-Calculus / Biology-for-MCAT) → categories/units → UMBRELLA keywords → SUBTOPIC keywords (as specific as possible). Every keyword has DESCRIPTION + EXAMPLES — the description+examples are embedded, NOT the keyword text. ACTION and REPRESENTATION keywords have NO umbrellas.

**Embedding & tagging — 7 EMBEDDINGS per problem:** 1 problem + 3 wrong-answer + 1 action + 1 representation + 1 prerequisite. Cosine match: run vs UMBRELLA keywords → top 3 umbrellas → inspect for closest keywords → top 15 keywords → LLM picks what's actually relevant + assigns weights 0–1 → normalize to sum 1. Action→ACTION pool, representation→REPRESENTATION pool, prerequisite/wrong-answer/problem→general topic keywords. End state: keyword weights for problem, each wrong answer, action, representation, prerequisite.
> Implemented: parts 1–8 + prerequisite (col `prerequisite_description{,_embedding}` on math/mcat_questions, migration `20260620000000`); enrichment in `lib/questionEnrichment.ts` (+ backfill `scripts/enrich-question-descriptions.ts`) generates+embeds+tags all 7. Tagging is currently cosine→nearest (LLM reweighting lives in admin `/api/tagging`, not the on-demand hot path — intentionally deferred to protect generation latency). Difficulty is a 0–1 number that DRIVES selection but is NEVER displayed to students (no Easy/Medium/Hard badges anywhere).

**Workflow:** Onboarding (brief demo → choose course(s), switchable + keeps progress → signup name/email/password/confirm OR login). Course home = homebase: profile reachable from ANY page (NavMenu); **auto/recommended is the MAIN attraction**; below it a topic map with lessons/flashcards/quizzes per topic + a general-practice option. Auto mode → asks to take a diagnostic OR skip & start from the beginning. Start-from-beginning → per-topic guided path **LESSON → FLASHCARDS → PRACTICE** (in course order; intro shown once per topic via `localStorage lodera_auto_intro_<course>`). The LESSON is stored, bite-sized, simple (can include graphs/tables); each page is followed by an in-lesson UNDERSTANDING-QUIZ that is NOT embedded and does NOT use the generation pipeline (just {problem, solution, choices+random correct index}); a wrong answer returns to the lesson page; pages navigate freely fwd/back. After all pages → PRACTICE from the general pool at a weight-targeted difficulty (escalate on correct, drop on wrong). Reaching the mastery threshold advances to the next subtopic. Practice INTERLEAVES past subtopics (spaced repetition); a past topic slipping below the re-review threshold (~0.75) is PRIORITIZED until back above it; a forgotten lesson triggers a REFRESHER (1–2 sentence/formula popup with a return-to-lesson option), then returns to the exact spot. DIAGNOSTIC: algorithmic, fewest problems, has an "I don't know" option, maps at UMBRELLA level with propagation up/down the prereq hierarchy, then hands off to auto mode IN COURSE ORDER (not weakest-first). MCAT auto follows the same lesson-first flow.

**Mastery model — logarithmic, threshold-advanced (2026-06-25, local-only):** Per-keyword mastery is a single 0–1 `score` on `*_student_keyword_states` shared by questions AND flashcards (universal state — the difficulty/flashcard-ratio logic reads from it). **All knobs live in ONE place: `apps/student/lib/courseEngine/adaptive.ts`** — `MASTERY_START` (0.30, fresh-keyword seed), `MASTERY_ADVANCE` (0.70, the advancement threshold), `BASE_GAIN`, `GAIN_CURVE` (the `(1−mastery)` exponent — the "logarithmic feel" knob), `DIFFICULTY_GAIN_WEIGHT` {easy/med/hard}, `SOURCE_WEIGHT` {question:1.0, flashcard:0.55}, `WRONG_BASE_PENALTY`, `WRONG_DIFFICULTY_WEIGHT` (missing an EASY item drops MORE), `DONT_KNOW_PENALTY`. The per-answer update is `updateMastery`/`updateMasteryMap`: **CORRECT** gain = `BASE_GAIN·(1−m)^GAIN_CURVE·diffGain·source·kwWeight` (fast early, asymptotically slow near 1 — harder + question = bigger); **WRONG** drop = `WRONG_BASE_PENALTY·m·diffMiss·source·kwWeight` (proportional to current mastery); **DON'T-KNOW**/skip = small `DONT_KNOW_PENALTY·m` drop. **The "4 correct in a row" gate is GONE** — a keyword is `mastered` purely when `score ≥ MASTERY_ADVANCE` (`isMastered`), in the server state machines (`/api/{math,mcat}/attempt` + `/flashcard-attempt`) AND the auto-page client advancement (mastery meter now shows "% to mastery" from the score, not a streak). `consecutive_correct` is still recorded for telemetry / next-question difficulty escalation, not gating. Future direction (not built): keep scoring past the threshold via spaced/harder questions to grow mastery toward 1.0 — the curve already supports it. **To tune: edit the constants in `adaptive.ts` and reload (HMR).**
- **5-distinct question batch:** `/api/{math,mcat}/next-question` generate a batch of **5** when they hold the gen-lock (serve 1, store 4; cold cells warm with 6), the generator's BATCH-DIVERSITY directive forces materially-different items, and a **near-duplicate dedup pass** (`filterNearDuplicates`, Jaccard ≥ `NEAR_DUP_THRESHOLD` on normalized stems, index-preserving) drops repeats before insert so stored items aren't the same task in different clothing.
- **Instant "Next" — DECIDE / BUFFER / FETCH / APPLY (2026-06-25, local-only):** the auto pages no longer block on a fresh round-trip per item. Serving is split four ways: **DECIDE** is the pure, instant adaptive choice (`decideNextDescriptor` → a `ServeDescriptor {keywordId, categoryId, kind, difficulty, forReview}`, same logic as the old `serveNextItem`); **BUFFER** is a shared client engine (`apps/student/lib/courseEngine/practiceBuffer.ts`, `createPracticeBuffer`, used by **both** `/math/[course]/auto` and `/mcat/auto`) that serves a matching descriptor from memory with **zero network**; **FETCH** runs only on a buffer miss; **APPLY** (`applyQuestion`/`applyFlashcards` — the ONLY place the seen/exclude/recent refs mutate) shows it. The next item is **prefetched during the answer-reveal window** (in `handleChoice`/`handleDontKnow` after the `/attempt` score lands, gated `!isReviewCard && !isMastered`) and **consumed** in `handleNext` (instant when settled). Both routes now return **`{ question, buffer: [...] }`** — the served item PLUS up to `BUFFER_MAX` (4) ready extras (the freshly generated/stored batch, or remaining stored top-N), so one fetch covers ~5 items and the just-generated extras become the next questions with no extra round-trip. `take`/`prefetch` are **side-effect-free** w.r.t. the page, so a discarded prefetch leaks nothing; the buffer is `clear()`ed on keyword advance (`handleNext`, `startKeyword`, `beginSkillIntro`) and on `QuestionToolbar`'s new **`onStateChange`** (fired after Prioritize, which changes selection inputs). Buffered extras carry the cheap max-weight `primary_keyword_id`; only the SERVED item pays for the precise `bestKeywordForQuestion` embedding lookup. **Server fetch made cheap too:** `loadTargetKeywords` (math + mcat) is wrapped in `serverCache.cached()` (5-min TTL — taxonomy is static, was re-paginated every request); and the next-question candidate query loads **light columns only** (no 4 embedding columns for the whole category) — embeddings are fetched in a second `.in('id', top-30)` query just for the MMR window (`mmrRerank` is a per-item rescale, so top-N is behaviour-equivalent).
- **LOCAL-ONLY workflow (current):** iterate on `npm run dev` → http://localhost:3002. **Do NOT deploy to prod** (no `vercel deploy`/push to `main`) until the user explicitly says so. Login on localhost requires the localhost origin in the **Supabase Auth redirect-URL allowlist** (dashboard → Authentication → URL Configuration) — see [[project-localhost-login-redirect]].

**Trust + Duolingo-mode work (2026-06-17, on `main`):**
- **Answer-keying contract:** generators emit the worked solution's final answer + distractors; `lib/assembleChoices.ts` (mirrored in admin) builds the 4 choices and picks the index in code, so the key can never contradict the solution. See [[project-answer-assembly-contract]]. Math/MCAT/learn generators all follow it.
- **Render trust:** `components/mcat/MathText.tsx` is the single math renderer — it normalizes ASCII science notation (`lib/scienceNotation.ts`: `Vmax`→`$V_{max}$`, `H2O`→`$H_2O$`), repairs JSON-escape LaTeX corruption (control chars `\f`/`\t`→`\frac`/`\theta`), renders bare LaTeX, and draws `<FunctionGraph>`/`<SlopeField>` viz segments.
- **Diagnostic → auto:** the math diagnostic shows a placement report then routes into **auto mode** (`/math/[course]/auto`). Auto mode is a **strict in-order GUIDED PATH** (rebuilt 2026-06-17): it walks the course's CORE categories in `order_index` order (precalc = all 11; **calc_ab = the 8 calc units only**, so it starts at Unit 1 Topic 1 = Limits — foundations are `role:foundation` and excluded from the forward path), one **topic (umbrella)** at a time in CED order. Per topic the sequence is **LESSON → FLASHCARDS → QUIZ** (the lesson+flashcards intro shows once per topic, remembered in `localStorage` key `lodera_auto_intro_<course>`); a topic completes only when all its in_depth skills are mastered, and only then is the next topic surfaced. **Spiral review** is interleaved (~35%) from already-learned skills of EARLIER topics — `/api/math/auto-plan` returns `frontier_topic` + ordered `next_focus` + `review_focus`, and the page builds the practice queue at WHOLE-COURSE scope so `review_pool` spans every earlier unit. A checkpoint quiz fires on **unit (category) completion**. **MCAT auto mode** (`/mcat/auto`, no diagnostic) walks each umbrella LESSON → FLASHCARDS → PRACTICE per skill; its **intro keyword** (`order_index = -1`, opens the umbrella) is **framing-only** — a look-ahead lesson + optional general flashcards (may be empty), then auto-mastered and skipped, **NEVER questioned** (`loadTargetKeywords({ excludeIntros: true })` drops `order_index = -1` from the `next-question`/`quiz` pools). See [docs/mcat-system.md](docs/mcat-system.md#intro-keywords--framing-only-order_index---1). `lib/mathTagging.ts loadTargetKeywords` filters by course (calc_ab asks calculus). The `/math/[course]` landing presents auto mode as a prominent **hero card** ("Recommended · Start here"); diagnostic + general practice are secondary links.
- **Honest progress:** `/api/{math,mcat}/taxonomy` returns `questions_answered`/`correct_answers` counted from the `*_question_attempts` log (NOT summed per-keyword, which 4×-inflated the old counters).
- **Onboarding:** a brand-new visitor can answer one free question at `/try` (public `/api/math/sample-question`) before the login wall; `LoginGate` has a Log in / Sign up toggle.
- **Question diversity:** `lib/questionDiversity.ts` caps consecutive same-keyword items + filters near-duplicate stems in the next-question/quiz routes.
- **Flashcards = memorization, NOT quizzes (2026-06-18):** the flashcard generators (`FLASHCARD_SYSTEM` in `lib/mathGenerator.ts` + `lib/mcatGenerator.ts`) produce pure recall cards — front = a CLOZE / fill-in-the-blank by default (a "?" question only when a cloze would be forced), back = the recalled nugget — never problem-solving or reasoning prompts (those are QUIZ). **Generation prompts are kept SHORT and example-free** (hardcoded examples over-fit and leak; fix a prompt by REDUCTION before adding). **All four generators (lesson / flashcard / quiz / refresher) are COVERAGE-FIRST — each must FULLY COVER the keyword's in-scope, balanced by combine-related + fold-derived-case-consequences + subject-based no-repeats.** Flashcard deck size is content-decided (no count target; `.slice(0,60)` runaway guard only); quizzes keep a soft `count` (hot-path latency); refreshers are terse BULLETS with no example (`rule_latex` newline-separated, `example_latex: ""`). See the `project_lesson_lab` memory for the full per-prompt rules. **MCAT flashcards follow the "mile wide, inch deep" depth standard — authoritative spec: [docs/mcat-depth-standard.md](docs/mcat-depth-standard.md)** (MilesDown-calibrated). This means DIRECTIONAL rules + approximate ranges + recognition/classification + named regulatory enzymes/cofactors + compartments — and DELIBERATELY **NO decimal side-chain $pK_a$, no exact $K_m$/$V_{max}$/$K_i$**. Round/memorable comparative values are kept (NADH=2.5 ATP vs FADH₂=1.5; glycolysis net 2 ATP) as are universal constants ($c$, $N_A$, 109.5°). The depth bar is encoded as `MCAT_DEPTH_RULE` in `lib/mcatGenerator.ts`. (Do NOT "fix" missing pKa decimals — that shallowness is intentional; content is deepened in BREADTH within each keyword's scope contract, the taxonomy has 934 in_depth keywords across 109 umbrellas after the 2026-06-26 full AAMC audit — see [docs/mcat-taxonomy-audit-2026-06.md](docs/mcat-taxonomy-audit-2026-06.md).) There is a standalone math flashcards page (`/math/[course]/[categoryId]/flashcards`); MCAT uses `/mcat/[categoryId]/flashcards`. **Flashcards are stored-preferred, so a prompt/format change requires deleting `*_flashcards` rows (and their `*_flashcard_attempts`/`mcat_flashcard_srs`, which FK with NO ACTION) so they regenerate.**

## Architecture
Turbo monorepo — two Next.js 15 (React 19) apps, three shared packages.
- **`apps/admin`** (:3001) — problem authoring, keyword management, RAG agent.
- **`apps/student`** (:3002) — student-facing precalc/AP Calc practice platform.

Shared packages (local workspace only — **never published to npm**):
- **`@ap-calc/types`** (`packages/types/src/index.ts`) — `Problem`, `ProblemInsert`, `TopicWeights`, `APCalcUnit`.
- **`@ap-calc/supabase`** (`packages/supabase/src/index.ts`) — singleton Supabase client.
- **`@ap-calc/constants`** (`packages/constants/`) — AP Calc topics, precalc keywords, unit→keyword map.

Both apps transpile shared packages via `transpilePackages` in `next.config.ts`.

If a key is set on Vercel but invalid, every AI feature 502s — check the key, not the code.

## Deployment
The student app **auto-deploys to Vercel production on push to `main`** (GitHub integration). The build is scoped to the student app (admin is excluded — it has pre-existing errors), **requires env vars set on the Vercel project**, and — because the git-based build only sees **committed** files — every source file must be committed (uncommitted routes/components break it even when it builds locally). Migrations are plain SQL in `supabase/migrations/`, applied manually in the Supabase SQL editor. Full details: **[docs/deployment.md](docs/deployment.md)**.

> **IRT difficulty calibration** writes `estimated_difficulty` on both `problems` and `rag_examples`. The `rag_examples` column is added by migration `20260609000000_rag_examples_estimated_difficulty.sql` — if it hasn't been applied to the live DB, `/demo` calibration degrades gracefully (resilient fallback) but won't converge. The `/demo` diagnostic is **test-like**: no per-question solution/answer reveal; the results screen auto-shows the score and offers a click-in review of right/wrong answers + solutions.
